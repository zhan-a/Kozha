# Prompt Injection Surface Audit (Prompt 17, step 1)

Walk-through of every place a user-controlled string reaches an LLM prompt
in the chat2hamnosys backend. Generated by auditing
`backend/chat2hamnosys/prompts/loader.py` consumers and every `"role":
"user"` / `"role": "system"` construction site.

The audit distinguishes two very different surfaces:

- **Template-interpolated fields** — values rendered into a prompt body by
  the Jinja2 loader (e.g. `{{ schema_block }}`). These can execute as
  Jinja2 if autoescape is off.
- **LLM message-content fields** — values placed into `{"role": "user",
  "content": ...}` and sent to the chat-completions API. These cannot
  execute as code but can still carry instruction-like content that the
  model may follow.

The loader uses `autoescape=False` and `StrictUndefined` (see
`backend/chat2hamnosys/prompts/loader.py:144-156`). Every entry below
names the binding, traces it back to its HTTP origin, and records the
sanitation status as of this audit.

---

## Jinja2 template-interpolated fields

All template variables below resolve to **module-level static constants**
(schema dumps, vocabulary tables, enum listings). None of them are
populated from an HTTP request. This is an architectural property worth
preserving — adding a user field to one of these render contexts would
break it.

| Prompt id | Version | Variable | Source | Current sanitation | Risk |
|---|---|---|---|---|---|
| `parse_description` | v1 | `schema_block` | `PartialSignParameters.model_json_schema()` dumped at import, `parser/description_parser.py:180` | N/A — system-generated JSON | **Low** — no user input |
| `parse_description` | v1 | `gap_fields_block` | Tuple constant `ALLOWED_GAP_FIELDS`, joined at import | N/A | **Low** |
| `parse_description` | v1 | `handshape_table`, `location_table`, `ext_finger_table`, `palm_table`, `movement_table`, `non_manual_table` | Module-level `dict[str, str]` constants (`HANDSHAPE_VOCAB` etc.) joined at import, `parser/description_parser.py:59-150` | N/A | **Low** |
| `parse_description` | v1 | `size_list`, `speed_list`, `repeat_list` | Module-level `tuple[str, ...]` constants | N/A | **Low** |
| `generate_clarification` | v1 | `allowed_fields_list`, `templates_block` | Tuple constant + hard-coded template library (`clarify/templates.py`) | N/A | **Low** |
| `interpret_correction` | v1, v2 | `allowed_scalars`, `allowed_nm`, `intents` | Enum / tuple constants from `correct/models.py` | N/A | **Low** |
| `generate_hamnosys_fallback` | v1 | _(none)_ | Template takes no context | N/A | **Low** |
| `generate_hamnosys_repair` | v1 | _(none)_ | Template takes no context | N/A | **Low** |

**Conclusion for template interpolation:** No user input flows into Jinja
context today. The loader's `autoescape=False` is acceptable *only under
that invariant*. See the regression test in `test_security_injection.py`
asserting no user-facing fields appear in any render context.

## LLM `user`-role message content

This is where user input actually meets the model. The backend puts user
prose, freeform corrections, and session payloads into the `user`
message. These are not Jinja-executed but are still the classic
prompt-injection channel.

| Prompt id | Field in user-message | HTTP origin | Current sanitation | Risk |
|---|---|---|---|---|
| `parse_description` | `prose` (string body, `parser/description_parser.py:355`) | `POST /sessions/{id}/describe` body `prose` (`api/models.py:DescribeRequest.prose`, `min_length=1`) | Pydantic min_length=1 only. **No length cap, no control-char stripping, no injection detection** prior to this audit. | **High** — primary attack surface |
| `generate_clarification` | `prose` field nested in JSON `user_payload` (`clarify/question_generator.py:288`, via `_build_user_payload` → last author turn text) | Derived from stored `description_prose` (same origin as above) and stored `ClarificationTurn.text` | Inherits sanitation from the describe step; none for follow-up answers. | **High** — flows the same prose + author answers |
| `generate_clarification` | `gaps[].reason`, `gaps[].suggested_question` in `user_payload` | Produced by the parser LLM; indirectly user-influenced via the prose it saw | Narrow JSON-schema output constraint | **Medium** — model-generated but user-shaped |
| `interpret_correction` | `correction.raw_text` in JSON `user_payload` (`correct/correction_interpreter.py:367`) | `POST /sessions/{id}/correct` body `raw_text` (`api/models.py:CorrectRequest.raw_text`, `min_length=1`) | Pydantic min_length=1 only. **No sanitation.** | **High** — second primary attack surface |
| `interpret_correction` | `correction.target_region` in `user_payload` | `POST /sessions/{id}/correct` body `target_region` (optional string, no length cap) | Pydantic type check only. | **Medium** — client-supplied label string |
| `interpret_correction` | `original_prose` in `user_payload` | Stored `description_prose` (origin: `prose`) | Inherits describe-step sanitation | **High** — re-flows user prose |
| `interpret_correction` | `parameters` snapshot in `user_payload` | Model-generated phonological struct | Schema-validated | **Low** — structured, typed |
| _(not a prompt template, but same channel)_ | `body.answer` string (`api/router.py:439`) | `POST /sessions/{id}/answer` body `answer` (str or int) | `.strip()` + non-empty check only | **High** — reaches clarifier prose context |

## Out-of-scope but noted

- **Session-level metadata** (`signer_id`, `display_name`, `domain`,
  `gloss`) is stored on `AuthorInfo` / `SignEntry` but never injected
  into a prompt. It is still PII and is addressed by the `pii_policy`
  config in §6 of the hardening plan.
- **HamNoSys output** from `generate_hamnosys_*` is structurally
  constrained (validated against the HamNoSys 4.0 grammar before
  storage) so the output channel is narrow.
- **Clarification questions and correction explanations** are
  natural-language strings shown to users — covered by the output
  moderation filter (§4 of the hardening plan).

---

## Risk summary

| Field | Pre-hardening risk | Target post-hardening |
|---|---|---|
| `DescribeRequest.prose` | High | Sanitized, length-capped, injection-screened, wrapped in `<user_description>` tags |
| `CorrectRequest.raw_text` | High | Sanitized, length-capped, injection-screened, wrapped in `<user_correction>` tags |
| `AnswerRequest.answer` (str) | High | Sanitized, length-capped, no injection screen (too short to warrant LLM call), wrapped at use site |
| `CorrectRequest.target_region` | Medium | Sanitized + length-capped (≤64 chars, enum-hint allowed) |
| Jinja template context variables | Low | Regression test locks the current invariant |

`backend/chat2hamnosys/security/` implements the defenses; the API
layer enforces them at the request boundary.
